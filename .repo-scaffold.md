# Repository Scaffold Structure

This document describes the standardized layout and entrypoint files for the money-trail project.

## Top-Level Directory Structure

```
money-trail/
├── .devcontainer/          # VS Code devcontainer configuration
│   ├── devcontainer.json   # Devcontainer configuration (references docker-compose.yml)
│   ├── Dockerfile          # Development container image
│   ├── docker-compose.yml  # Dev workspace services
│   └── devcontainer-lock.json
├── .github/                # GitHub workflows and configuration
├── .airflow/               # Local Airflow metadata (gitignored)
├── compose/                # Runtime compose configuration documentation
├── dags/                   # Apache Airflow DAGs
├── dbt/                    # dbt project files
├── data/                   # Data artifacts (gitignored)
├── docs/                   # Architecture decisions and guides
├── scripts/                # Bootstrap and helper scripts
├── sql/                    # SQL analysis and QA queries
├── tests/                  # Test files
├── Dockerfile              # Production container image
├── docker-compose.yml      # Production/local runtime services
├── requirements.txt        # Python dependencies
├── .env.example            # Environment variable template
├── .env                    # Local environment variables (gitignored)
├── README.md               # Project overview
├── CONTEXT.md              # Domain glossary and architecture context
├── LICENSE                 # Project license
└── .gitignore              # Git ignore patterns
```

## Entrypoint Files

### Development Container Setup
- **`.devcontainer/devcontainer.json`**: Main entry point for VS Code devcontainer
  - References `.devcontainer/docker-compose.yml` (relative path)
  - Runs `postCreateCommand: bash scripts/bootstrap.sh` for initialization
  - Configures VS Code extensions and settings

### Container Images
- **`.devcontainer/Dockerfile`**: Development container image
  - Based on Microsoft devcontainers Python 3.12
  - Includes development tools (docker, git, ripgrep, fd, sqlite3, etc.)
  - Installs Python dependencies via uv

- **`Dockerfile`** (root): Production container image
  - Based on python:3.12-slim-bookworm
  - Multi-stage build (builder + runtime)
  - Includes Airflow runtime environment

### Docker Compose Configurations

#### `.devcontainer/docker-compose.yml` (Development)
- **Service**: `workspace` (VS Code development container)
- **Context**: `..` (root of repo)
- **Dockerfile**: `.devcontainer/Dockerfile`
- **Environment**: SQLite-based Airflow, local development settings
- **Volumes**: Full repo mount, Docker socket for container-in-container

#### `docker-compose.yml` (Production/Local Runtime)
- **Services**:
  - `postgres`: PostgreSQL metadata database
  - `airflow-init`: Initialization service
  - `airflow-webserver`: Airflow web UI (port 8080)
  - `airflow-scheduler`: Airflow DAG scheduler
- **Context**: `.` (root of repo)
- **Dockerfile**: `Dockerfile`
- **Environment**: PostgreSQL-backed Airflow, production-like settings

### Bootstrap and Setup
- **`scripts/bootstrap.sh`**: Entrypoint script for devcontainer initialization
  - Creates required directories (.airflow, logs, data/)
  - Initializes Airflow metadata database (when BOOTSTRAP_AIRFLOW=1)
  - Creates default admin user (devadmin)
  - Sets up dbt profiles

- **`scripts/start_airflow.sh`**: Helper for starting Airflow services
  - Starts scheduler and webserver (for non-containerized local development)

### Configuration Templates
- **`.env.example`**: Template for environment variables
  - FEC_SAMPLE_URL, OPENLINEAGE_URL, AIRFLOW__CORE__FERNET_KEY
  - Copy to `.env` to customize local environment

## Container Reference Resolution

### Development Workflow
1. Open repo in VS Code → Reopen in Container
2. VS Code reads `.devcontainer/devcontainer.json`
3. Loads `.devcontainer/docker-compose.yml` (relative path)
4. Builds `.devcontainer/Dockerfile` from repo root context
5. Runs `postCreateCommand: bash scripts/bootstrap.sh` inside container
6. Workspace ready for development

### Production/Runtime Workflow
1. Execute `docker-compose up -d` (from repo root)
2. Reads `docker-compose.yml` (root)
3. Builds `Dockerfile` (root)
4. Brings up Airflow services (postgres, airflow-webserver, airflow-scheduler)
5. Services accessible at http://localhost:8080

## Path Consistency

| Component | Dev Compose | Prod Compose | Notes |
|-----------|-------------|--------------|-------|
| Build context | `..` (root) | `.` (root) | Both resolve to repo root |
| Dockerfile | `.devcontainer/Dockerfile` | `Dockerfile` | Different images for dev vs prod |
| App mount | `/workspaces/money-trail` | `/app` | Dev uses VS Code path, prod uses container path |
| DAGs folder | `/workspaces/money-trail/dags` | `/app/dags` | Referenced in environment variables |
| Data directory | `/workspaces/money-trail/data` | `/app/data` | Mounted volumes from repo |

## Adding New Entrypoints

To maintain consistency when adding new scripts or configurations:

1. **Bootstrap scripts**: Add to `scripts/` directory
2. **Compose services**: Update relevant docker-compose.yml with proper context
3. **Docker images**: Keep Dockerfile in same directory as its docker-compose.yml reference
4. **Environment setup**: Add variables to `.env.example` with clear documentation
